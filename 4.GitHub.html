2. Introduccion a la Terminal

Comandos básicos:

  1. pwd: Ver ruta actual.
  2. mkdir: Crea una carpeta (mkdir nombre_carpeta).
  3. touch: Crear archivos (touch nombre_archivo).
  4. rm: Borrar un archivo o carpeta (rm archivo_a_borrar). Cuidado con este comando, puedo borrar incluso mi disco duro.
  5. cat: Ver contenido de un archivo (cat archivo_a_leer).
  6. ls: Muestra los archivos de la carpeta actual.
  7. ls -a: Muestra absolutamente todos los archivos.
  8. ls -al: Ver todos los archivos.
  9. ls -l: Ver todos los archivos como una lista.
  10. cd: Navegar entre carpetas.
  11. cd /: Ir a la ruta principal
  12. cd o cd ~ : Ir a la ruta de tu usuario.
  13. cd ..: Regresa una carpeta.
  14. history: Historial de comandos ejecutadis.
  15. !732: Ejecutaría el comando numero 732 mostrado en history.
  16. clear o Ctrl + L: Limpia la terminal.
  17. Tecla Tab: Muestra predicciones posibles de carpetas o comandos.
  18. --help: Descubre todos los argumentos de un comando (cat --help).

3. Iniciar un repositorio

Iniciar un repositorio en git: (git init) este comando se encargará de 2 cosas:

  1. Crea una carpeta .git que guarda toda la base de datos con cambios atómicos del proyecto.
  2. Crea un área Staging que guarda temporalmente mis archivos y permitirá más adelante, guardar estos cambios en el repositorio.

3.1 Estados de los archivos en Git:

  1. Lectura: Click

3.2 Comandos para mover archivos entre los estados de Git:

  1. git status: ve el estado de mis archivos / carpetas.
  2. git add: muevo archivos de Untracked / Unstaged al Staged.
  3. git add nombre_archivo: agrega archivo / carpeta.
  4. git add -A: agrega todos los archivos del proyecto (Untrackeds o unstageds).
  5. git reset HEAD: saca archivos del estado Staged para devolverlos a su anterior estado.
  6. git commit: mueve archivos de Unstaged a Tracked.
  7. git rm --cached: mueve los archivos que le indiquemos al estado Untracked.
  8. git rm --force: elimina los archivos de git y del disco duro. (git guarda el registro de la existencia de los archivos, por lo que puedo recuperarlos si es necesario, pero debo usar comandos más avanzados).

4. Ramas y Merge

  1. Git es una base de datos muy precisa que tiene los cambios y crecimiento de mi proyecto.
  2. Los commits son la única forma de tener un registro de los cambios, todos los commits se aplican sobre una rama.
  3. Por defecto inicio en la rama master y creo nuevas ramas a partir de esta, para crear flujos de trabajo independientes.
  4. Los equipos de desarrollo tienen un estándar, todo lo que esté en la rama master va a producción.
  5. Las nuevas características / experimentos van en la rama “development” y los errores se solucionan en la rama “hotfix”, estas ramas se unen a master cuando estén listas
  6. Crear una nueva rama se le llama Checkout y unir dos ramas se llama Merge.
  7. Puedo crear todas las ramas y commits que quiera.
  8. Ten en cuenta que combinar ramas puede generar conflictos, algunos archivos pueden ser diferentes en ambas ramas.

5. Analizar cambios en los archivos

  1. git show nombre_archivo: muestra los cambios que han existido sobre un archivo.
  2. git commit: un commit vacío no es permitido, esto abre la consola de vim.
  3. Esc + i: para insertar texto en vim y escribir un commit.
  4. git log nombre_archivo: no me acuerdo para que era.
  5. Shift + z + z: sale del editor vim, y guarda mi mensaje
  6. git diff commit_X commit_Y: muestra la diferencia entre dos commits.

6. Volver en el tiempo en mi repositorio

  1. git reset algun_commit_antiguo --soft: regresa a la version de ese commit antiguo.
  2. git reset algun_commit_antiguo --hard: todo vuelve al estado de ese commit antiguo, y borra los commits que se hicieron desde ese en adelante.
  3. git log --stat: ver cambios a partir del commit
  4. git checkout hash_de_un_commit nombreArchivo: Pasa a los cambios de ese commit, puede ser uno antiguo, pero aun no estan borrados los cambios mas recientes, a no ser que hagamos un commit desde el que estamos situados;
  5. git checkout master nombreArchivo Regresa al commit actual de mi proyecto.
  6. letra q: para salir de (END)

6.1 Git reset & Git rm

  1. git rm --cached: Elimina los archivos del área de Staging y del próximo commit pero los mantiene en mi disco duro.
  2. git rm --force: Elimina los archivos de Git y del disco duro. Git siempre guarda todo, por lo que puedo acceder al registro de la existencia del archivo, puedo recuperarlo (pero debo usar comandos avanzados).
  3. git reset --soft: Borra todo el historial y los registros de Git, pero guardo cambios que tenga en Staging, así puedo aplicar las últimas actualizaciones a un nuevo commit.
  4. git reset --hard: Borra todo. comando peligroso
  5. git reset HEAD: Saca archivos del Staging para que los últimos cambios de estos archivos no se envíen al último commit, a menos que yo cambie de opinión y los incluya de nuevo en staging con git add.
  6. Caso de uso de git reset HEAD: suponga que hago git add pero me doy cuenta que un archivo no debe subirse aún
  7. para solucionarlo, debo sacar ese archivo de Staging para hacer commit de todos los demás, git reset HEAD mueve los cambios de Staging a Unstaged, sigo teniendo los últimos cambios del archivo, el repositorio mantendrá el archivo (no con sus últimos cambios pero sí con los últimos en los que hicimos commit) y no habré perdido nada.

7. Ramas o branches de Git

  1. git commit -am "mensaje_commit": solo sirve con archivos que alguna vez tuvieron un git add.
  2. git commit -a: lo mismo que arriba pero dirige al editor vin para escribir el mensaje.
  3. git branch nombre_de_la_rama: Crea una nueva rama, puedo verificarlo con git show
  4. git checkout nombre_de_la_rama: me cambia a la que escriba
  5. git checkout -b "nombre_de_la_rama": crea rama y pasa a ella.
  6. git branch: ver ramas existentes.
  7. git merge nombre_de_otra_rama: fusiona 2 ramas. Debo ubicarme en la rama que consideraré principal, para pegar a ella la rama secundaria, luego del merge podré ver los commits de las dos ramas juntas.

8. Resolución de conflictos al hacer merge

  1. Cuando uso git merge o git checkout, estoy cambiando de rama o creando un nuevo commit, no borrando ramas o commits (recuerda: borra commits (git reset) y ramas (git branch -d)).
  2. Git puede resolver algunos conflictos automáticamente: cambios, nuevas líneas, etc. Pero a veces no sabe cómo resolver las diferencias, Ej: 2 ramas distintas hacen cambios distintos en la misma línea.
  3. Este conflicto se resuelve manualmente, haciendo el merge, ir al editor de código y elegir si quiero quedarme con alguna de las 2 versiones o algo diferente. Editores como VSCode ayudan a resolverlos sin necesidad de borrar o escribir líneas de texto.
  4. Recuerda que siempre debe crear un nuevo commit para aplicar los cambios del merge. Si Git puede resolver el conflicto hará commit automáticamente.
  5. Los archivos con conflictos por el comando git merge entran en un estado conocido como Unmerged. Muy parecidos a los archivos en Unstaged, similar a un estado intermedio entre Untracked y Unstaged, solo ejecute git add para pasarlos al staging y git commit para aplicar los cambios.

10. Configura tus llaves SSH en local

  1. En la terminal escribo ssh-keygen -t rsa -b 4096 -C "mi_email@gmail.com"
  2. -t: algoritmo a usar para crear la llave, rsa es el algoritmo mas popular.
  3. -b: que tan compleja sera la llave, 4096 es la complejidad de la llave desde una perspectiva matemática.
  4. -C: a que correo electronico estará conectada la llave.
  5. Aparecerá un mensaje de: generendo llave, y luego:
  6. Enter file in which to save the key (/c/Users/Admin/.ssh/id_rsa): ruta por defecto donde guardará la llave, al dar enter la guardará, pero se puede cambiar si quiero.
  7. Enter passphrase (empty for no passphrase): Para poner una contraseña, al dar enter la deja vacía.
  8. Verifica si el proceso corre con eval $(ssh-agent -s) esto muestra algo como: Agent pid 1111; Ó ssh-add ~/.ssh/id_rsa
  9. El proceso crea dos llaves, una publica y una privada, agrego la privada: ssh-add ~/nombre_llave_priv , y listo

15. Pull Requests en GitHub

  1. Pull request: Funcionalidad de github en la que un colaborador pide que revisen sus cambios antes de hacer merge a una rama. Al hacer un pull request se genera una conversación que pueden seguir los demás usuarios del repositorio, así como autorizar y rechazar los cambios.
  2. Flujo del pull request:
  3. Se trabaja en una rama paralela con los cambios deseados (git checkout -b rama)
  4. Se hace commit a la rama (git commit -am 'Comentario')
  5. Se suben al remoto los cambios (git push origin rama)
  6. En GitHub se hace el pull request comparando la rama master con la rama del fix.
  7. Uno, o varios colaboradores revisan que el código sea correcto y dan feedback (en el chat del pull request)
  8. El colaborador hace los cambios que desea en la rama y lo vuelve a subir al remoto.
  9. Se aceptan los cambios en GitHub
  10. Se hace merge a master desde GitHub
  11. (Cuando se modifica una rama, también se modifica el pull request)

16. Forks

  1. Forks / Bifurcaciones: Característica de GitHub donde se crea una copia exacta del estado actual de un repo directamente en GitHub, este repo podrá servir como otro origen y se podrá clonar como cualquier otro.
  2. Es una bifurcación del repo completo, tiene una historia en común, pero de repente se bifurca y pueden variar los cambios, ya que ambos proyectos podrán ser modificados en paralelo y para estar al día un colaborador tendrá que estar actualizando su fork con la información del original.
  3. Al hacer fork de un proyecto en GitHub, me convierto en dueño del repo fork, puedo trabajar en este con todos los permisos, es un repo completamente diferente que el original, teniendo alguna historia en común.
  4. Son importantes porque es la manera en la que funciona el open source, una persona puede no ser colaborador de un proyecto, pero puede contribuír.
  5. Al hacer un fork, GitHub sabe que se hizo el fork del proyecto, por lo que se le permite al colaborador hacer pull request desde su repo propio.
  6. Cuando trabajo en un proyecto que existe en diferentes repos remotos (a causa de un fork) es probable que desee trabajar con ambos repos, para ésto puedes crear un remoto adicional desde consola.
  7. git remote add nombreDelRemoto urlDelRemoto
  8. git remote upstream https://github.com/ freddier/hyperblog
  9. Al crear un remoto adicional podré, hacer pull desde el nuevo origen (si tienes permisos puedes hacer fetch y push)
  10. git pull remoto rama
  11. git pull upstream master
  12. Este pull traerá los cambios del remoto, por lo que estará al día en el proyecto, el flujo de trabajo cambia, en adelante se estará trabajando haciendo pull desde el upstream y push al origin para pasar a hacer pull request.
  13. git pull upstream master
  14. git push origin master

19. Git Rebase reorganiza el trabajo realizado (mala practica)

  1. rebase reescribe la historia del repositorio, cambia la historia de donde comenzó la rama y solo debe ser usado de manera local.
  2. Me ubico en la rama que quiera, ahí es donde traeré información de otra rama. Ej: en rama_1 escribo git rebase rama_2 para copiar la informacion de rama_2
  3. luego si quiero, puedo pasar a rama_2 y hacer git rebase rama_1, para traer la info de esa rama.
  4. Al ver la historia de los commits, aparecerá que mi rama_1 ha existido, pero ha tenido siempre la misma informacion de rama_2, es como si no hubiera pasado nada fuera de la rama rama_2,
  5. PERO, en realidad pudimos haber hecho cambios en la otra rama.
  6. git branch -D rama_1 borraria esa rama que antes usamos, pero ya mi informacion esta en rama_2, asi que "no paso nada"

20. Git Stash Guardar cambios en memoria y recuperarlos

  1. Stashed: lista de estados que guarda cambios que hizo en Staging para poder cambiar de rama sin perder el trabajo que aún no guardo en un commit. Permite cambiar de ramas y trabajar en otras cosas y más adelante retomar el trabajo que tenía en Staging.
  2. útil pues a veces no se permite cambiar de rama porque tengo cambios sin guardar, y no siempre es un buen cambio como para hacer un commit, pero no quiero perder ese código.
  3. git stash: guarda el trabajo actual del Staging en una lista temporal llamada Stash, y podré recuperarla en el futuro.
  4. Puede poner un mensaje en el stash para diferenciarlos en el git stash list: git stash save "mensaje"
  5. Obtener elementos stash: El stashed se comporta como una Stack de datos tipo LIFO (Last In, First Out), así puedo acceder al método pop.
  6. git stash pop: recupera y saca de la lista el último estado del stashed y lo inserta en el staging area, así que es importante saber en qué rama me encuentro para poder recuperarlo, pues el stash es agnóstico a la rama o estado, siempre recuperará los cambios que hice en el lugar que lo llamo.
  7. Aplicar cambios de un stash específico y eliminarlo: git stash pop stash@{< numero_stash >}
  8. Retomar los cambios de una posición específica del Stash: git stash apply stash@{< numero_stash >}
  9. Ver el numero de un stash, ver la lista de cambios guardados y poder recuperarlos o hacer algo con ellos: git stash list
  10. Para crear una rama y aplicar el stash mas reciente: git stash branch < nombreDeLaRama >
  11. Si deseo crear una rama y aplicar un stash específico: git stash branch nombre_de_rama stash@{< numero_stash >}
  12. Usar esos comandos creará una rama con el nombre < nombreDeLaRama >, me pasará a ella y tendré el stash especificado en el staging area.
  13. Para eliminar los cambios más recientes dentro del stash (elemento 0): git stash drop
  14. Si conozco el índice del stash que quiero borrar: git stash drop stash@{< num_indice__stash >}
  15. Si deseo eliminar todos los elementos del stash: git stash clear

20.1 Consideraciones:

  1. Crear un stash tomará los archivos que han sido modificados y eliminados. Para que tome un archivo creado es necesario agregarlo al Staging Area con git add [nombre_archivo] con la intención de que git tenga un seguimiento del archivo, o también usando el comando git stash -u (guardará en stash los archivos que no estén en staging).
  2. El cambio más reciente (al crear un stash) SIEMPRE recibe el valor 0 y los que estaban antes aumentan su valor.
  3. Al aplicar un stash este no se elimina, es buena práctica eliminarlo.

21. Git Clean limpiar mi proyecto de archivos no deseados

  1. git clean: borra documentos duplicados, (debo usarlo con otro atributo).
  2. git clean --dry-run: ver que archivos iré a borrar.
  3. git clean -f: borra esos archivos
  4. git ignora carpetas, las carpetas copiadas debo borrarlas a mano.
  5. imagenes o archivos que estan en .gitignore no se borrarán.
  6. El parametro -d ayuda a borrar carpetas untracked. Ej: git clean -df

22. Reconstruir commits

  1. Si hago un commit, pero olvido agregar un cambio, debo hacer el git add de los cambios, y luego uso git commit --amend.

23. Buscar en archivos y commits

  1. git grep color: busca en mi proyecto, los archivos en donde está la palabra color.
  2. git grep -n color: saldrá un output donde dice en qué línea está lo que buscamos.
  3. Con git grep -c color: saldrá un output el cual dirá cuántas veces se repite esa palabra y en qué archivo.
  4. git grep -c "<p>": cuántas veces uso el atributo p de HTML.
  5. git log-S “cabecera”: cuantas veces use la palabra cabecera en todos los commits.
  6. grep –> para los archivos, log –> para los commits
  7. Use el git log -S “palabra”: trae las palabras parecidas a las que estamos buscando.
  8. Es mejor usar git log --all --oneline | grep “cabecera”, trae los commits en donde se encuentra la palabra.

24. Comandos y recursos colaborativos en Git y GitHub

  1. git shortlog -sn: muestra cuantos commits ha hecho cada miembros del equipo.
  2. git shortlog: muestra mensajes de los commits
  3. git shortlog -sn --all: muestra cuantos commits hizo cada miembro del equipo, incluso los que han sido eliminados
  4. git shortlog -sn --all --no-merge: muestra cuantos commit han hecho cada miembros quitando los eliminados sin los merges
  5. git blame nombrearchivo: muestra quien hizo cada cosa, linea por linea
  6. git blame -c nombrearchivo: es como el comando de arriba, pero con identación
  7. git nombredelcomando --help: muestra como funciona ese comando.
  8. git blame nombrearchivo -L35,50: muestra quien hizo cada cosa linea por linea indicándole desde que linea ver ejemplo: -L35,53, tambien le puedo agregar el '-c' al final
  9. git branch -r: muestra todas las ramas remotas
  10. git branch -a: muestra todas las ramas tanto locales como remotas
  11. git log --oneline Muestra el id y título del commit.
  12. git log --decorate: Muestra donde se encuentra el head point en el log.
  13. git log --stat: Explica el número de líneas que se cambiaron brevemente.
  14. git log -p: Explica el número de líneas que se cambiaron y muestra que se cambió en el contenido.
  15. git shortlog: Indica que commits ha realizado un usuario, muestra el usuario y el titulo de sus commits.
  16. git log --pretty=format:"%cn hizo un commit %h el dia %cd": Muestra mensajes personalizados de los commits.
  17. git log -3: Limitamos el número de commits.
  18. git log --after=“2018-1-2”
  19. git log --after=“today”
  20. git log --after=“2018-1-2” --before=“today”: Commits para localizar por fechas.
  21. git log --author=“Name Author”: Commits realizados por autor que cumplan exactamente con el nombre.
  22. git log --grep=“INVIE”: Busca los commits que cumplan tal cual está escrito entre las comillas.
  23. git log --grep=“INVIE” –i: Busca los commits que cumplan sin importar mayúsculas o minúsculas.
  24. git log – index.html: Busca los commits en un archivo en específico.
  25. git log -S “Por contenido”: Buscar los commits con el contenido dentro del archivo.
  26. git log > log.txt: guardar los logs en un archivo txt
  27. Si borro un archivo, y doy git checkout -- . me regresa al ultimo commit guardado
